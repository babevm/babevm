void bvm_stacktrace_populate ( bvm_throwable_obj_t *  throwable_obj)

Given an bvm_throwable_obj_t with a populated backtrace, create its array of java.lang.StackTraceElement objects. This function will have no effect if the array is already populated, or the throwable has no backtrace.

After creating the array, the throwable's backtrace is NULLed and the memory freed.

Parameters
throwable_objthe throwable to populate.

◆ bvm_stacktrace_populate_backtrace()

void bvm_stacktrace_populate_backtrace ( bvm_throwable_obj_t *  throwable_obj)

Create a stack backtrace structure for a given throwable object. This function allocates memory for a bvm_stack_backtrace_t with enough room in its frames array to hold a bvm_stack_backtrace_frameinfo_t for each frame on the stack (starting at the top - the current executing method).

The backtrace ignores any methods on the stack that are from the java.lang.Throwable class or any of its subclasses. Why? This backtrace is called during construction of an exception object way up at the 'Throwable' class constructor. So as we're executing this native code, the stack is actually full of constructor methods for each Throwable subclass down to the actual thrown exception class' constructor. Simply put, we just ignore them all when creating the backtrace.

The actual backtrace population is implemented as a stack visit callback - #populate_backtrace_callback.

Parameters
throwable_objthe throwable object

◆ bvm_stacktrace_print_to_console()

void bvm_stacktrace_print_to_console ( bvm_throwable_obj_t *  throwable_obj)

Prints a throwable's stack trace to the console. If the throwable has no backtrace data this function has no effect. As the java level stack trace printing removes a throwable's backtrace - this function only does a stacktrace if the throwable has not printed a stack trace at the java level.

The logic here is effectively a re-implementation of the java level stacktrace.

Parameters
throwable_objthe throwable to print the trace of.
